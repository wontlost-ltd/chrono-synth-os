# 0032 — PPF v1 uses zod schemas, not JSON Schema

**Status:** Accepted
**Date:** 2026-Q2 (Phase 4-A)
**Scope:** `packages/contracts/src/ppf/`, `docs/ppf/v1/spec.md`

## Context

The Portable Persona Format (PPF) v1 is the JSON-LD interchange document
for moving a persona between Chrono Synth runtimes (and, eventually,
between vendors). Its schema is the contract that anyone implementing
PPF must validate against — both us and external implementers.

We had to choose how to express the schema. The two mainstream options:

- **zod** — TypeScript-first, schemas are also runtime parsers, no codegen
  step, schemas live next to the types they describe.
- **JSON Schema** (ajv / jsonschema) — vendor-neutral, language-agnostic,
  the de-facto standard for cross-language interchange, well-supported
  in Python / Go / Java code generators.

JSON Schema is the obvious "interop" answer. We chose zod anyway.

## Decision

**PPF v1 schemas are authored in zod** in `@chrono/contracts`.

We export the typed `PpfV1DocumentSchema` and friends; the round-trip
contract test under `src/test/contract/` validates real persona
documents against them. The human-readable spec in `docs/ppf/v1/spec.md`
documents fields, ordering, and JSON-LD `@context`; external implementers
read the spec, not the zod source.

For external implementers who need a machine-readable schema, we ship a
JSON Schema export generated from zod via `zod-to-json-schema`. That
export lives at `docs/ppf/v1/schema.json` and is regenerated by a script,
not authored directly.

## Consequences

**Wins**

- Schemas are runtime parsers. `PpfV1DocumentSchema.parse(json)` both
  type-narrows and validates in one call. No `if (!validator(json))`
  branch.
- Schemas double as the source of types: `type PpfV1Document = z.infer<typeof PpfV1DocumentSchema>`.
  No drift between TS types and validator.
- Authoring is short. A 200-field schema fits in ~80 lines of zod; the
  same in JSON Schema is ~400 lines and fights the IDE.
- Refinements (cross-field constraints, e.g. `values.length === values_order.length`)
  are first-class in zod via `.refine()`. JSON Schema needs custom
  keywords or a separate validator.

**Costs**

- Non-TS implementers have to consume the JSON Schema export, not the
  source. They miss the comments and refinement helpers.
- We carry one extra dev-dep (`zod-to-json-schema`) and a script to
  regenerate the export when zod schemas change. If the regeneration
  drifts (script broken, no one notices), interop docs go stale.
  Mitigation: contract test re-runs the regeneration in CI and diffs
  against the committed export.
- zod runtime cost is small but non-zero (~40 KB minified). Not a
  concern for a backend; not relevant for SPA either since zod is
  already a transitive dep there.

## Alternatives considered

- **JSON Schema as source of truth, generate TS types via `json-schema-to-ts`:**
  rejected. Generated types are notoriously ugly (deep `Tail<...>`-style
  combinator spam) and drift the moment you write a refinement that
  doesn't have a JSON Schema equivalent. The ergonomic tax fell on
  every kernel author writing persona data.

- **Two separate authorings (zod for TS, JSON Schema for export), kept
  in sync by review discipline:** rejected. We'd already lived through
  this pattern with the old `@chrono/contracts` v0.x types and contract
  schemas; drift was inevitable inside two PRs. Single source + generated
  export is the only sustainable path.

- **Protobuf / Avro:** rejected for the wire format. PPF is JSON-LD by
  product requirement (linked-data semantics, namespaced via `@context`).
  Binary formats fight that goal.

## How to enforce going forward

- New PPF fields are added to the zod schema first; the JSON Schema
  export is regenerated and committed in the same PR.
- The contract test in `src/test/contract/ppf-v1-test-vectors.test.ts`
  parses every test vector through zod. Adding a vector that doesn't
  fit the schema fails CI.
- External implementer reports of "your JSON Schema export disagrees
  with the spec" are P1 bugs against the regeneration script, not
  against the spec.
